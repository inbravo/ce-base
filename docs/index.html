TECHNICAL SOLUTION DESIGN · AWS

Neptune-Backed Knowledge Graph for Delivery Context

How a developer would actually build the "How does this all fit together?" check — from raw tables to a Bedrock Agent answer.

1

Problem statement & scope

The Bedrock Agent handling Sarah's questions needs to reason across an order, a courier scan, a delivery hub, and A-to-Z Guarantee eligibility — facts that today live as flat, unrelated records in Redshift and S3. Answering "is it on its way" or "does this qualify for a refund" requires a human to write a new ad-hoc SQL join per question, with no shared, checkable definition of what these facts mean together.

IN SCOPE
  • Neptune-backed graph modeling Order / Scan / Hub / Guarantee
  • Ingestion path from existing Redshift / S3 sources
  • Python query/API layer, callable by a Bedrock Agent action group
  • Two implementation options for the "meaning" layer
  • Baseline observability
OUT OF SCOPE
  • PBAC / row-level security
  • Golden Gatekeeper review workflow
  • Infrastructure-as-code / deployment automation
  • Cost optimization
  • Changes to upstream courier or order systems
2

Architecture overview

Source data is transformed and loaded into Neptune, which a Python query layer running inside the same VPC accesses on behalf of the Bedrock Agent.

3

Graph data model

Nodes: Order(order_id, order_date), Scan(scan_id, scan_time, scan_type), Hub(hub_id, location), Guarantee(guarantee_type, eligibility_rule_ref).
Edges: SCANNED_AT (Order→Scan), LOCATED_AT (Scan→Hub), TRIGGERS (Scan→Guarantee, when scan_type='DAMAGED').

OPTION A

Property graph

"Class" is just a vertex label. TRIGGERS edges are pre-computed at ingestion time based on scan_type.

  • No real inference — a new event type that should trigger the guarantee needs an ETL update
  • Query with Gremlin or openCypher, interchangeably, on the same dataset
  • Simpler to operate, faster to query
OPTION B

RDF / OWL

Entities are URIs. rdf:type assigns OWL classes (e.g. :Scan9931 rdf:type :DamagedOnArrivalEvent).

  • Ontology declares :DamagedOnArrivalEvent rdfs:subClassOf :GuaranteeTriggeringEvent
  • SPARQL can match any subclass of GuaranteeTriggeringEvent — real class inference
  • Requires its own Neptune cluster, queryable only via SPARQL
⚠ Gremlin and openCypher share one property-graph dataset in Neptune — RDF does not. You cannot mix RDF and property-graph data in the same cluster, and RDF is only queryable via SPARQL. Pick one model per cluster; running both means two deployments.

Example instance — Order #48213 (property-graph model)

This is one instance, not the schema — a different scan with scan_type: IN_TRANSIT would have no TRIGGERS edge at all, since that edge only gets written when the ingestion logic (or OWL subclass match, in Option B) decides this scan qualifies.

4

Ingestion pipeline

Batch: a Glue ETL job reads the three Redshift tables, computes edges (or emits triples), and writes to S3; Neptune's built-in Bulk Loader ingests from there.
Incremental: a small Lambda on the scan-event stream writes new vertices/edges (or triples) directly, for near-real-time updates.

📄 Where the policy documents fit: "Fulfilment Policy §4.2" and "Ops Policy §2.1" aren't data rows — they're the rules that decide which edges or classes get created. That logic is hand-encoded into the ETL transform (Option A) or the OWL ontology's class axioms (Option B) by whoever owns ingestion. This design covers the mechanics, not who reviews those rules.
5

Query / API layer — Python, both options

Same authentication either way: IAM SigV4 signing plus VPC security groups; the Lambda must be VPC-attached to reach the cluster.

OPTION A · GREMLIN (gremlinpython) 
from gremlin_python.driver import client, serializer

conn = client.Client(
    'wss://my-cluster.cluster-xxxx.us-east-1.neptune.amazonaws.com:8182/gremlin',
    'g',
    message_serializer=serializer.GraphSONSerializersV2d0()
)

# walk Order → Scan → Hub → Guarantee
query = """
  g.V().has('Order', 'order_id', '48213')
    .out('SCANNED_AT').as('scan')
    .out('LOCATED_AT').as('hub')
    .out('TRIGGERS').hasLabel('Guarantee')
    .valueMap()
"""
result = conn.submit(query).all().result()
conn.close()
OPTION A · OPENCYPHER (neo4j driver over Bolt) 
from neo4j import GraphDatabase

driver = GraphDatabase.driver("bolt://my-cluster...:8182", auth=None)  # IAM/VPC, not basic auth

with driver.session() as session:
    result = session.run("""
        MATCH (o:Order {order_id:'48213'})-[:SCANNED_AT]->(s:Scan)
              -[:LOCATED_AT]->(h:Hub)-[:TRIGGERS]->(g:Guarantee)
        RETURN o, s, h, g
    """)
    for record in result:
        print(record)

3

Ontology — auto-extracted, not hand-modeled

Unlike Neptune, there's no explicit schema step where a developer declares Order, Scan, Hub, and Guarantee as node types and wires up SCANNED_AT / TRIGGERS edges. Genie Ontology continuously mines signals and infers the graph:

SIGNALS IT MINES

What Genie observes

  • Table and column names, and foreign-key-like naming patterns (order_id, hub_id)
  • Join patterns seen across the SQL query history
  • Which dashboards and queries are used most, and by whom
WHAT A DEVELOPER CONTROLS

The only levers available

  • Unity Catalog column comments and table/column tags
  • Marking an asset "Certified" — certified sources are weighted higher
  • Which queries and dashboards get promoted, since usage itself is a signal
⚠ There's no query language for hand-writing a TRIGGERS edge. If Genie hasn't seen enough certified, authoritative usage connecting Scan.scan_type='DAMAGED' to a guarantee outcome, it won't reliably infer that relationship. The fix is exposing that logic in a governed, certified asset — a Metric View or a certified query (Section 5) — not in application code, and not in a hand-written graph edge either.
4

Ingestion pipeline

Lakeflow Declarative Pipelines ingest raw carrier scan events and order events into Bronze → Silver → Gold Delta tables inside Unity Catalog — standard Delta ingestion, nothing graph-specific happens here. The difference from Neptune's pipeline: there's no discrete "compute edges" step. What the Gold table owner attaches as column comments, tags, and certification is the only thing that later biases what Genie Ontology treats as an authoritative relationship.

📄 Where the policy documents fit: "Fulfilment Policy §4.2" and "Ops Policy §2.1" aren't encoded as ETL transform logic here, because there's no explicit edge-creation step to encode them into. In practice this rule still has to live somewhere a human maintains it — most reliably as a certified Metric View or certified query (Section 5). Left undone, it lives nowhere formal at all, which is a governance gap Genie Ontology's auto-extraction doesn't close by itself.

Architecture overview

Delta tables are ingested and governed through Unity Catalog. Genie Ontology continuously mines table/column metadata, query history, and dashboard usage to infer an entity graph, weighting sources by authority. A Genie Space queries both the Ontology and a governed Metric View to answer Alex, over the Genie Conversation API.

3

Ontology — auto-extracted, not hand-modeled

Unlike Neptune, there's no explicit schema step where a developer declares BalanceCheck, FraudCheck, and CardCheck as node types and wires up a TRIGGERS edge to a decline reason. Genie Ontology continuously mines signals and infers the graph:

SIGNALS IT MINES

What Genie observes

  • Table and column names, and foreign-key-like naming patterns (txn_id, card_id)
  • Join patterns seen across the SQL query history
  • Which dashboards and queries are used most, and by whom
WHAT A DEVELOPER CONTROLS

The only levers available

  • Unity Catalog column comments and table/column tags
  • Marking an asset "Certified" — certified sources are weighted higher
  • Which queries and dashboards get promoted, since usage itself is a signal
⚠ There's no query language for hand-writing a TRIGGERS edge. If Genie hasn't seen enough certified, authoritative usage connecting expiry_date < today to a decline outcome, it won't reliably infer that relationship. The fix is exposing that logic in a governed, certified asset — a Metric View or a certified query (Section 5) — not in application code, and not in a hand-written graph edge either.
4

Ingestion pipeline

Lakeflow Declarative Pipelines ingest raw payments, fraud-score, and card-network events into Bronze → Silver → Gold Delta tables inside Unity Catalog — standard Delta ingestion, nothing graph-specific happens here. The policy documents (Overdraft Policy, Risk Model Card, ISO 8583) are loaded as files into Unity Catalog Volumes, where Genie can read and weigh them as sources — but nothing forces Genie to connect a specific document to a specific check.

📄 Where the policy documents fit: acct_balance, risk_score, and expiry_date are plain Delta columns — but whether they pass is decided by three separate documents sitting in a Unity Catalog Volume. Nothing automatically links "Overdraft Policy §2.1" to the BALANCE CHECK table the way a hand-written GOVERNED_BY edge would in Neptune. In practice this link still has to live somewhere a human maintains it — most reliably as a certified Metric View or certified query (Section 5). Left undone, it lives nowhere formal at all, which is a governance gap Genie Ontology's auto-extraction doesn't close by itself.

TECHNICAL SOLUTION DESIGN · AWS

Neptune-Backed Decision Chain for Decline Reasons

How a developer would actually build the three-check decision trace — from raw tables and policy documents to a Bedrock Agent answer.

1

Problem statement & scope

The Bedrock Agent handling Alex's questions needs to reason across a balance check, a fraud score, and a card-status check — each one a fact governed by its own database and its own policy document, evaluated in a fixed order. Answering "why was my card declined" today requires a human to manually check three different systems and two different policy documents, with no shared, checkable definition of what these facts mean together or which order they're evaluated in.

IN SCOPE
  • Neptune-backed graph modeling the balance / fraud / card-status decision chain
  • Ingestion path from the payments DB, fraud engine, and card network feeds
  • How the two policy documents (Overdraft Policy, Risk Model Card) and one standard (ISO 8583) become graph-side thresholds and mappings
  • Python query/API layer, callable by a Bedrock Agent action group
  • Baseline observability
OUT OF SCOPE
  • PBAC / row-level security
  • Golden Gatekeeper review workflow
  • Infrastructure-as-code / deployment automation
  • Cost optimization
  • Changes to upstream fraud model or card network systems
2

Architecture overview

Source data and policy documents are transformed and loaded into Neptune, which a Python query layer running inside the same VPC accesses on behalf of the Bedrock Agent.

3

Graph data model

Nodes: Transaction(txn_id, status), BalanceCheck(acct_balance, result), FraudCheck(risk_score, result), CardCheck(expiry_date, result), PolicyDocument(doc_id, title, rule_quote), DeclineReason(code, description).
Edges: CHECKED_BY (Transaction→BalanceCheck), NEXT_CHECK (chains BalanceCheck→FraudCheck→CardCheck in evaluation order), GOVERNED_BY (each check node → the one PolicyDocument that decided its result — Overdraft Policy §2.1, Risk Model Card v3.2, or ISO 8583 Field 39), TRIGGERS (CardCheck→DeclineReason, when result='EXPIRED').

Every check node carries two inputs, matching the use case's diagram exactly: a structured property read from a database (acct_balance, risk_score, expiry_date), and a GOVERNED_BY edge to the unstructured policy document that decides what counts as a pass.

OPTION A

Property graph

"Class" is just a vertex label. TRIGGERS edges are pre-computed at ingestion time from each check's result.

  • No real inference — a new check type needs an ETL update
  • Query with Gremlin or openCypher, interchangeably, on the same dataset
  • Simpler to operate, faster to query
OPTION B

RDF / OWL

Entities are URIs. rdf:type assigns OWL classes (e.g. :Check9931 rdf:type :CardExpiredEvent).

  • Ontology declares :CardExpiredEvent rdfs:subClassOf :DeclineTriggeringEvent
  • SPARQL can match any subclass of DeclineTriggeringEvent — real class inference
  • Requires its own Neptune cluster, queryable only via SPARQL
⚠ Gremlin and openCypher share one property-graph dataset in Neptune — RDF does not. You cannot mix RDF and property-graph data in the same cluster, and RDF is only queryable via SPARQL. Pick one model per cluster; running both means two deployments.

Example instance — Transaction #TXN-58217 (property-graph model)

Matches the use case's diagram exactly: the same txn_id, the same three DB field values (£340 / 12 of 100 / 2026-06-30), and the same policy references — just drawn as a graph instead of a flowchart.

This is one instance, not the schema — a card check with result: PASSED would have no TRIGGERS edge at all, since that edge only gets written when the ingestion logic (or OWL subclass match, in Option B) decides this check qualifies.

4

Ingestion pipeline

Batch: a Glue ETL job reads the payments, ledger, and fraud-score tables, computes each check's result, the NEXT_CHECK chain, and the GOVERNED_BY edges (or emits triples), and writes to S3; Neptune's built-in Bulk Loader ingests from there.
Incremental: a small Lambda on the transaction-event stream writes new vertices/edges (or triples) directly, for near-real-time updates.

📄 Where the structured and unstructured sources meet: acct_balance, risk_score, and expiry_date are plain database columns — but whether they pass is decided by three separate documents, each ingested as its own PolicyDocument node: "Overdraft Policy §2.1" (a threshold rule), "Risk Model Card v3.2" (a scoring cutoff), and the ISO 8583 Field 39 standard (a code mapping). The GOVERNED_BY edge linking a check to its document is hand-encoded into the ETL transform (Option A) or the OWL ontology's class axioms (Option B) by whoever owns ingestion. This design covers the mechanics, not who reviews those rules.

from gremlin_python.driver import client, serializer

conn = client.Client(
    'wss://my-cluster.cluster-xxxx.us-east-1.neptune.amazonaws.com:8182/gremlin',
    'g',
    message_serializer=serializer.GraphSONSerializersV2d0()
)

# walk Transaction → BalanceCheck → FraudCheck → CardCheck → DeclineReason
query = """
  g.V().has('Transaction', 'txn_id', 'TXN-58217')
    .out('CHECKED_BY')
    .repeat(out('NEXT_CHECK')).until(out('TRIGGERS').hasNext().or(outE('NEXT_CHECK').count().is(0)))
    .out('TRIGGERS').hasLabel('DeclineReason')
    .valueMap()
"""
result = conn.submit(query).all().result()
conn.close()
